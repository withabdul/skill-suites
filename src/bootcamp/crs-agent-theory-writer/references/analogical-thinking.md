# Analogical Thinking

Create real-world analogies that make abstract technical concepts click. Every analogy should make the reader say "oh, that's what it means" — not "huh, that's confusing."

## What Success Looks Like

Every analogy illuminates without oversimplifying. The technical concept remains accurate; the analogy provides a mental model, not a replacement. A good analogy maps one structural similarity without implying false parallels.

## Your Approach

### Analogy Quality Criteria

A good analogy meets all three:
1. **Structural match** — the analogy shares the key mechanism with the concept
2. **Familiar domain** — the reader already understands the analogy's source
3. **No false implication** — the analogy doesn't imply things that are false about the concept

### Analogy Construction Process

1. **Identify the core mechanism** — what's the one thing that makes this concept click?
2. **Find a familiar parallel** — what everyday experience works the same way?
3. **Map the parallels** — explicitly state what maps to what
4. **State the limits** — where the analogy breaks down
5. **Write it naturally** — embed in prose, don't isolate as "Analogy Time"

### Example Quality Spectrum

**Poor analogy** (oversimplified, false implication):
> "React hooks itu seperti Velcro — nempel dan lepas dengan mudah."
→ Structural match is weak. Velcro implies temporary, which is misleading about state.

**Good analogy** (structural match, familiar, honest about limits):
> "React hooks itu seperti stiker yang kamu tempel di buku catatan. Stiker-nya nempel di halaman tertentu, dan kalau halamannya sobek, stiker-nya ikut hilang. Tapi — ini batas analoginya — stiker bisa dipindah ke halaman lain, sementara state di React nggak berpindah sendiri."

### When Analogies Are Required

- Beginner material: **every new concept** gets at least one analogy
- Intermediate: **complex concepts only** (simple ones can be explained directly)
- Advanced: **paradigm shifts only** (the reader already knows the domain)
